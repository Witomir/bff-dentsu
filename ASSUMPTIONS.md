## Project assumtpions
1. The downstream product service does not support filtering/sorting/offsetting/pagination. The backend will fetch all the products and do the filtering/sorting/offsetting/pagination in memory. We fetch all the products at once. Fetching such a huge list of products will decrease perforormance, one the cache is invalidated, but it could be improved later by periodically refreshing the cache in the background, instead on demant with request.
2. The list of products fetched from a downstream service contains all the products that are needed to be displayed in the UI. No additional fetching is needed to get the details of the products.
2. The possible filter values are hardcoded now in the backend. If we wanted to make them dynamic, there should be a different architecture running on the backend, like a background job that fetches the products, extracts the possible filter values and stores them in a memory.
3. I took the liberty to change the sorting param convention, by using the Pageable standard from Spring Boot, that requires sorting in format `&sort=price,desc` instead of `&sort=price&order=desc`. I think it is better for future extensibility, as it allows to sort by multiple fields, like `&sort=price,desc&sort=name,asc`. It could be discussed with the fronted developers.
4. The mobile/web client switch is done by a header `X-Client-Type` with possible values `mobile` and `web`.
